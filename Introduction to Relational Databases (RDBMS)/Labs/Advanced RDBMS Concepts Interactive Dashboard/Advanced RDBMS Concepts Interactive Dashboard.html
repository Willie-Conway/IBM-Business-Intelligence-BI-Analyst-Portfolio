Functional Dependency

X → Y: One Attribute Determines Another

Definition

A relationship where the value of one attribute (X) uniquely determines the value of another attribute (Y).

// Notation: X → Y
// EmployeeID → EmployeeName
// If you know EmployeeID, you can determine EmployeeName

EmployeeID | EmployeeName | Department
----------|--------------|-----------
101 | Alice | HR
102 | Bob | IT
103 | Charlie | Finance

Properties:

  • Reflexivity: X → X (every attribute determines itself)
  • Transitivity: If X → Y and Y → Z, then X → Z
  • Augmentation: If X → Y, then XZ → YZ

Real-world Example:

In a student database: StudentID → StudentName, StudentEmail

Knowing the StudentID (101) uniquely determines the student's name (Alice) and email.

Multi-Valued Dependency

X →→ Y: One Attribute Determines a Set of Values

Definition

A relationship where one attribute (X) determines a set of possible values for another attribute (Y).

// Notation: X →→ {Y1, Y2, ...}
// EmployeeID →→ {ProjectID, ProjectName}

EmployeeID | ProjectID | ProjectName
----------|-----------|------------
101 | P1 | Project Alpha
101 | P2 | Project Beta
102 | P1 | Project Alpha
103 | P3 | Project Gamma

Key Characteristics:

  • Represents many-to-many relationships
  • Violates 4th Normal Form (4NF)
  • Requires separate tables to eliminate redundancy

Real-world Example:

In a course registration system: StudentID →→ {CourseID, CourseName}

A student can enroll in multiple courses, creating multiple rows for the same student.

Candidate Keys

Minimal Sets that Uniquely Identify Rows

Definition

A minimal set of attributes that uniquely identifies each row in a relation.

// Candidate Keys for a Student Table:

StudentID | Email | Phone | Name
101 | alice@edu.com | 123-456-7890| Alice
102 | bob@edu.com | 234-567-8901| Bob
103 | charlie@edu.com| 345-678-9012| Charlie

Candidate Keys in this table:

  • StudentID - Unique for each student
  • Email - Should be unique per student
  • Phone - Should be unique per student

Properties:

  • Uniqueness: Each combination must identify exactly one row
  • Minimality: No proper subset should be a key
  • Irreducibility: Cannot remove any attribute and still maintain uniqueness
Database Design Best Practices

Optimal RDBMS Design Principles

🎯
1. Normalize to Reduce Redundancy

Follow 1NF, 2NF, 3NF rules to eliminate data duplication and ensure consistency.

⚡
2. Index for Performance

Create indexes on frequently queried columns and foreign keys for faster retrieval.

🔗
3. Establish Foreign Key Relationships

Use foreign keys with proper cascading actions to maintain referential integrity.

📊
4. Choose Appropriate Data Types

Use the smallest data type that fits your data to optimize storage and performance.

🏗️
5. Plan for Scalability

Consider partitioning, sharding, and archiving strategies for future growth.

RDBMS Design Best Practices

Professional Database Design Guidelines

🎯 Before Design

  • Understand business requirements
  • Identify entities and relationships
  • Determine data constraints
  • Plan for future growth

⚡ Performance

  • Normalize for integrity
  • Denormalize for speed
  • Index strategically
  • Partition large tables

🔧 Maintenance

  • Regular backups
  • Monitor performance
  • Update statistics
  • Archive old data

📊 E-commerce Database Example

-- Best Practice Implementation:

-- 1. Normalized Design
CREATE TABLE Customers (
  CustomerID INT PRIMARY KEY,
  Email VARCHAR(255) UNIQUE NOT NULL,
  Name VARCHAR(100) NOT NULL
);

-- 2. Foreign Key Relationships
CREATE TABLE Orders (
  OrderID INT PRIMARY KEY,
  CustomerID INT REFERENCES Customers(CustomerID),
  OrderDate DATE NOT NULL,
  INDEX idx_customer (CustomerID)
);

-- 3. Strategic Indexing
CREATE INDEX idx_order_date ON Orders(OrderDate);
CREATE INDEX idx_customer_email ON Customers(Email);
